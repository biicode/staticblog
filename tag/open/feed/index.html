]]> Today we are introducing the open command. If you’ve already played a bit with biicode you have seen how easy is to reuse code that other people have published: You just write the #include directive (or require, or import… dependending on your programming language), and the code gets downloaded into your deps folder.

Now we are going to talk about how to continue improving code that you have already published. There are two possible scenarios here: If you still have the hive where your published your block from, you can keep editing it there and carry on publishing new versions. But what if you formatted your computer, or want to publish from Bahamas and you didn’t bring your laptop with you? Here is where the open command comes in handy.

The open command gets a block from biicode servers and places the code into your blocks folder. It will get the last version or the version you’re using if the block is among your dependencies. Once the block is open you can edit it and publish new versions. In other words, opening a block means that its source code is available for edition inside the blocks folder of a local hive. As simple as that!

Let’s see it with an example:

$ cd my_project
$ tree blocks
blocks
0 directories, 0 files
$ bii open sparkfun/rf_links
Opened sparkfun/rf_links
INFO: Saving files on disk
INFO: Retrieving resources from server
INFO: Retrieving resources from server
INFO: Computing dependencies
INFO: Saving dependences on disk
$ tree blocks
blocks
└── sparkfun
   └── rf_links
       ├── readme.md
       ├── rflink_rx.cpp
       └── rflink_tx.cpp
2 directories, 3 files

As you can see, all source files fo the rf_links block of user sparkfun are automatically downloaded into the blocks folder of your hive. Now one might ask a few questions:

  • Can I open a block owned by another user? Yes, you can. You can open any published block and make your modifications and improvements locally in your hive.

  • Can I publish my modifications for an opened block? Yes, but they will be published to a new branch <your_user_name>/<branch_name>. This new branch will be accessible from your own profile, browsing the biicode web, and other users will need to edit their policies to be able to use your version of the block. We will talk about branching in more detail in a future post.

Finally, you might find interesting these resources in the biicode documentation, and continue learning:

  • The open command reference. Describes this command and use cases.
  • Project layout. Describes the general folders layout for your biicode project. Each one of them, contains a deps folder, with your code dependencies, and a blocks folder, containing the code you are currently working on.
